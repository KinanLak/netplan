# Realtime Collaboration Architecture

Status: proposed target architecture for replacing the current Convex PR implementation.

Audience: an implementation agent with no prior conversation context.

The goal is to make Netplan feel like a polished Figma/Google Docs style multiplayer editor: instant local interactions, no visual flicker, robust multi-user convergence, deterministic undo/redo, and no regressions from the local Zustand version.

## Executive Decision

Use Convex as the realtime server-authoritative backend, but do not use raw Convex document ids as the editor's domain ids and do not call Convex mutations directly from many UI components.

Build a document session layer with these properties:

- Stable application object ids generated by the client before any optimistic UI is shown.
- A single `MapDocumentProvider` per active floor/document.
- A pure operation engine: `visibleDocument = applyPendingOperations(serverDocument, pendingOperations)`.
- A sequential, idempotent client outbox.
- Server-side validation for all invariants.
- Undo/redo implemented as new normal operations, not as fragile special-case mutation replay.
- Presence kept separate from durable document state.

This is intentionally close to Figma's model for visual editors: centralized server authority, client-generated ids, property/object operations, last-writer-wins at property boundaries where appropriate, and local pending operations layered over acknowledged server state to avoid flicker.

## Why The Current PR Is Not Robust Enough

The current `convex` branch moves durable state from local Zustand to Convex, which is the right strategic direction. However, the implementation shape introduces correctness and UX risks.

Known issues in the current PR:

- `useMapCommands(currentFloorId)` is instantiated in multiple places: `FlowCanvas`, `Toolbar`, `DeviceDrawer`, route content, and `useWallToolSession`. Each instance owns its own refs for history ordering and pending wall ids. That means undo ordering and mutation sequencing can depend on which component issued a command and when the network ACK resolves.
- Optimistic device creation uses fake Convex ids. If the user drags or deletes the new device before Convex returns the real `_id`, subsequent mutations can target an invalid id.
- Undo/redo depends on backend ids that may change after recreate. This forces remapping and still leaves edge cases when other users delete or edit referenced objects.
- The UI uses Convex query results directly with optimistic update callbacks. Convex optimistic updates are replayable and temporary, so generated values inside them are dangerous unless they are already stable mutation arguments.
- Selection and presence can keep stale references after remote deletes.
- The first-run dataset regresses from the previous rich mock map to mostly empty buildings/floors unless seed data is expanded.
- Server validation is incomplete for collision and map invariants.

These are not isolated bugs. They come from the architecture. Patch only if absolutely necessary, but the intended fix is to replace the sync architecture.

## Product Requirements

Required UX qualities:

- Local edits must feel instant.
- No object should jump backward while waiting for server ACK.
- Creating an object and immediately dragging, selecting, linking, deleting, or undoing it must work.
- Two clients editing different objects must never block each other.
- Two clients editing different properties of the same object should not conflict unnecessarily.
- Two clients editing the same property should converge deterministically.
- Undo/redo must be local to the user's own actions and must not corrupt later edits from other users.
- Remote deletes must not leave broken UI state.
- Connection loss must degrade gracefully and recover by resyncing.
- Presence must feel live but must never be required for document correctness.

Non-negotiable correctness requirements:

- Domain ids used by the editor must be stable before the first local render.
- Every client operation must be idempotent.
- Every operation must be safe to retry.
- The server must be authoritative for persisted document state.
- The client must be authoritative only for pending local projection and ephemeral UI state.
- Server snapshots plus pending local operations must produce the visible canvas state.
- Server rejects must be handled explicitly and visibly enough for a polished UX.
- Durable document tables must not depend on localStorage.

## Research Notes

Figma's public writeups are directly relevant because Netplan is a visual object editor, not a text editor.

Useful takeaways from Figma's multiplayer architecture:

- Figma uses a client/server model with a central authority rather than fully decentralized CRDT state.
- Each object has a stable id and a collection of properties.
- Property changes can be treated like last-writer-wins registers ordered by the server.
- Creation requires client-generated unique ids so newly created objects can be edited immediately, including offline or before ACK.
- Deletion removes server state, while undo stores the deleted object's snapshot in the local undo buffer and restores it later as a new operation.
- Local changes are applied immediately; incoming server changes that conflict with unacknowledged local changes must not cause flicker.
- Undo in multiplayer must generate redo history at undo time, not blindly replay old snapshots.

Useful takeaways from Convex docs:

- Convex queries are reactive and update after mutations.
- `withOptimisticUpdate` applies temporary local query updates while a mutation is pending.
- Optimistic update callbacks must be synchronous.
- Optimistic updates can run more than once because Convex may replay them when local query results change.
- Therefore, optimistic update callbacks must be deterministic. Generate ids/op ids before invoking the mutation and pass them as args.
- `useConvexConnectionState` can be used to expose connection health and pending request UX.

Yjs is not the recommended primary architecture for this project right now:

- Yjs is excellent for collaborative text, rich documents, and offline CRDT sync.
- Netplan needs queryable structured data, server validation, collision rules, links, future auth/permissions, and Convex integration.
- A full Yjs document would make server-side invariant validation and Convex querying more complex.
- If future requirements include deep offline editing across long sessions, revisit Yjs or a dedicated CRDT layer. Do not introduce it for the current map editor unless the product requirement changes.

## Domain Model

Separate database ids from domain ids.

Convex `_id`:

- Internal database identity.
- Used only inside Convex functions and low-level persistence code.
- Never used in React Flow node ids, links, undo history, presence, shortcut logic, selection state, or public domain commands.

Application object ids:

- Stable ids generated by the client before dispatching an operation.
- Used everywhere in the app domain.
- Persisted as unique fields in Convex tables.
- Referenced by links, presence, undo history, and operations.

Suggested ids:

```ts
type ClientId = string;
type OperationId = string;
type DeviceObjectId = string;
type WallObjectId = string;
type LinkObjectId = string;
type FloorObjectId = string;
type BuildingObjectId = string;
```

Recommended format:

```ts
`${clientId}:${counter}`;
```

or UUIDv7 if a package already exists or is deliberately added. Prefer the `clientId:counter` format because it is easy to debug and guarantees uniqueness per client if the counter is persisted with the identity.

The `netplan-identity` localStorage entry should contain:

```ts
interface LocalIdentity {
  clientId: string;
  sessionId: string;
  nextObjectCounter: number;
  nextOperationCounter: number;
  displayName: string;
  colorHue: number;
}
```

If counters are not persisted, use random UUIDs. Do not use `Date.now() + Math.random()` for durable ids.

## Convex Schema Direction

The exact schema can evolve, but the durable tables should contain `objectId` fields and unique indexes where needed.

Example shape:

```ts
buildings: defineTable({
  objectId: v.string(),
  name: v.string(),
  orderKey: v.string(),
}).index("by_object_id", ["objectId"]);

floors: defineTable({
  objectId: v.string(),
  buildingId: v.string(),
  name: v.string(),
  orderKey: v.string(),
})
  .index("by_object_id", ["objectId"])
  .index("by_building", ["buildingId", "orderKey"]);

devices: defineTable({
  objectId: v.string(),
  floorId: v.string(),
  type: deviceType,
  name: v.string(),
  hostname: v.optional(v.string()),
  position,
  size,
  metadata: deviceMetadata,
  deletedAt: v.optional(v.number()),
  updatedAt: v.number(),
  updatedBy: v.string(),
})
  .index("by_object_id", ["objectId"])
  .index("by_floor", ["floorId"]);

walls: defineTable({
  objectId: v.string(),
  floorId: v.string(),
  start: position,
  end: position,
  color: wallColor,
  geometryKey: v.string(),
  deletedAt: v.optional(v.number()),
  updatedAt: v.number(),
  updatedBy: v.string(),
})
  .index("by_object_id", ["objectId"])
  .index("by_floor", ["floorId"])
  .index("by_floor_geometry", ["floorId", "geometryKey"]);

links: defineTable({
  objectId: v.string(),
  floorId: v.string(),
  fromDeviceId: v.string(),
  fromPort: v.optional(v.string()),
  toDeviceId: v.string(),
  toPort: v.optional(v.string()),
  label: v.optional(v.string()),
  deletedAt: v.optional(v.number()),
  updatedAt: v.number(),
  updatedBy: v.string(),
})
  .index("by_object_id", ["objectId"])
  .index("by_floor", ["floorId"])
  .index("by_from_device", ["fromDeviceId"])
  .index("by_to_device", ["toDeviceId"]);

clientOperations: defineTable({
  opId: v.string(),
  clientId: v.string(),
  clientSeq: v.number(),
  floorId: v.optional(v.string()),
  kind: v.string(),
  status: v.union(v.literal("applied"), v.literal("rejected")),
  result: v.optional(v.any()),
  error: v.optional(v.string()),
  createdAt: v.number(),
  appliedAt: v.number(),
})
  .index("by_op_id", ["opId"])
  .index("by_client_seq", ["clientId", "clientSeq"]);
```

Avoid `v.any()` in normal app types. If Convex validators force it for operation storage, isolate it in the operation log and decode to typed operations at the boundary.

## Document Snapshot

Define a single canonical client-side document snapshot type independent of Convex rows:

```ts
interface MapDocumentSnapshot {
  floorId: FloorObjectId;
  devices: Array<Device>;
  walls: Array<WallSegment>;
  links: Array<Link>;
}
```

Domain objects use application ids:

```ts
interface Device {
  id: DeviceObjectId;
  floorId: FloorObjectId;
  type: DeviceType;
  name: string;
  hostname?: string;
  position: Position;
  size: Size;
  metadata: DeviceMetadata;
}

interface WallSegment {
  id: WallObjectId;
  floorId: FloorObjectId;
  start: Position;
  end: Position;
  color: WallColor;
  geometryKey: string;
}

interface Link {
  id: LinkObjectId;
  floorId: FloorObjectId;
  fromDeviceId: DeviceObjectId;
  fromPort?: string;
  toDeviceId: DeviceObjectId;
  toPort?: string;
  label?: string;
}
```

Do not leak Convex `Doc<"devices">` into canvas or domain code. Convert at the repository/session boundary.

## Operation Model

Operations are the source of local changes and server mutations.

Base shape:

```ts
interface OperationMeta {
  opId: OperationId;
  clientId: ClientId;
  clientSeq: number;
  createdAt: number;
}
```

Concrete operations:

```ts
type MapOperation =
  | { kind: "device.create"; meta: OperationMeta; device: Device }
  | {
      kind: "device.patch";
      meta: OperationMeta;
      deviceId: DeviceObjectId;
      patch: DevicePatch;
    }
  | { kind: "device.delete"; meta: OperationMeta; deviceId: DeviceObjectId }
  | { kind: "link.create"; meta: OperationMeta; link: Link }
  | { kind: "link.delete"; meta: OperationMeta; linkId: LinkObjectId }
  | { kind: "walls.add"; meta: OperationMeta; walls: Array<WallSegment> }
  | { kind: "walls.delete"; meta: OperationMeta; wallIds: Array<WallObjectId> }
  | { kind: "batch"; meta: OperationMeta; operations: Array<MapOperation> };
```

Patch types should be explicit:

```ts
interface DevicePatch {
  name?: string;
  hostname?: string;
  position?: Position;
  size?: Size;
  metadata?: DeviceMetadata;
}
```

Do not use broad object merging with `any` or untyped partial documents.

## Pure Operation Engine

Create a pure engine module, suggested path:

```txt
src/map-engine/applyOperation.ts
src/map-engine/applyOperation.test.ts
src/map-engine/buildInverseOperation.ts
src/map-engine/buildInverseOperation.test.ts
src/map-engine/validation.ts
src/map-engine/validation.test.ts
```

Required functions:

```ts
export function applyOperation(
  snapshot: MapDocumentSnapshot,
  operation: MapOperation,
): ApplyOperationResult;

export function applyOperations(
  snapshot: MapDocumentSnapshot,
  operations: ReadonlyArray<MapOperation>,
): ApplyOperationResult;

export function buildInverseOperation(
  snapshotBeforeOperation: MapDocumentSnapshot,
  operation: MapOperation,
): MapOperation | null;
```

`ApplyOperationResult` should include both the next snapshot and any non-fatal issues:

```ts
interface ApplyOperationResult {
  snapshot: MapDocumentSnapshot;
  applied: boolean;
  reason?: ApplyOperationReason;
}
```

Rules:

- Applying an operation twice should be safe whenever possible.
- Deleting a missing object is a no-op with a reason, not a crash.
- Patching a missing object is a no-op with a reason, not a crash.
- Creating an object with an existing id should either be idempotent if payload matches or rejected if payload conflicts.
- Wall additions must deduplicate by `geometryKey` within a floor.
- Link creation must require existing endpoints in the visible snapshot.
- Link deletion must be idempotent.
- Device deletion must remove links connected to that device in the visible snapshot.
- Batch operations must be atomic from a UX/history perspective. If strict atomic server behavior is not implemented yet, the local engine still treats batch as one history item.

The operation engine must not import React, Convex, Zustand, or React Flow.

## MapDocumentProvider

Create a single provider per active floor/document. Suggested path:

```txt
src/map-session/MapDocumentProvider.tsx
src/map-session/useMapDocument.ts
src/map-session/outbox.ts
src/map-session/materializeDocument.ts
src/map-session/mapRepository.ts
```

Provider responsibilities:

- Subscribe to Convex server snapshot queries for the active floor.
- Keep `pendingOperations` in memory for operations dispatched by this client and not yet acknowledged.
- Materialize `visibleDocument` by applying pending operations over the latest server snapshot.
- Expose stable command functions to UI components.
- Push undo history when a local operation is accepted into the local pending queue, not when the server resolves.
- Send operations to Convex through a single outbox in client order.
- Mark operations as acknowledged or rejected based on server response.
- On rejection, remove the pending operation, rematerialize, and show a user-safe error state/toast if the action was visible.
- Expose connection state and pending save state.

Suggested API:

```ts
interface MapDocumentSession {
  floorId: FloorObjectId | null;
  document: MapDocumentSnapshot;
  isReady: boolean;
  isSaving: boolean;
  hasRejectedOperations: boolean;
  connectionState: "connecting" | "connected" | "disconnected";
  dispatch: (operation: MapOperation, options?: DispatchOptions) => void;
  commands: MapDocumentCommands;
  undo: () => void;
  redo: () => void;
}
```

UI components must not call Convex mutations directly for map domain changes. They call session commands.

Only one `MapDocumentProvider` instance should own the outbox and undo/redo stacks for the current floor. If a component needs data, it consumes the provider. Do not recreate the command layer inside `Toolbar`, `FlowCanvas`, `DeviceDrawer`, or wall hooks.

## Materialized View Anti-Flicker Rule

The visible document must always be derived as:

```ts
const visibleDocument = applyOperations(
  serverDocument,
  pendingOperations,
).snapshot;
```

This is the core anti-flicker invariant.

Example:

1. Server says device A is at x=10.
2. Client drags A to x=100 and sends pending op.
3. Server sends an older acknowledged snapshot with x=10.
4. UI still shows x=100 because pending op is reapplied.
5. When the server ACK includes x=100, pending op is removed and UI remains x=100.

Do not set UI state directly from `useQuery(api.devices.listForFloor)` for the canvas. Use the materialized document.

## Outbox

The outbox sends operations in local order.

Suggested behavior:

- `dispatch(operation)` appends to `pendingOperations` immediately.
- If not already flushing, start flush.
- Flush one operation at a time to preserve user order.
- On success, remove the operation from pending by `opId`.
- On duplicate success, remove it from pending by `opId`.
- On validation rejection, remove it from pending and record a rejected operation notice.
- On network failure, keep it pending and retry when connected.
- Never reorder operations to optimize throughput until correctness is proven.

This is intentionally conservative. Device drag commits are one operation per completed drag, so sequential flush should be fast enough.

## Convex Mutations

Prefer a single mutation endpoint for document operations:

```ts
api.mapOperations.apply;
```

Arguments:

```ts
{
  operation: MapOperationInput;
}
```

Server flow:

1. Decode and validate operation shape.
2. Check `clientOperations.by_op_id`.
3. If already applied, return stored result.
4. If already rejected, return stored rejection.
5. Validate domain invariants against current DB state.
6. Apply DB writes transactionally within the mutation.
7. Insert `clientOperations` row with status and result.
8. Return result.

Important server validation:

- Floor exists.
- Object ids are unique on create.
- Device operations target an existing non-deleted device on the expected floor.
- Link endpoints exist, are non-deleted, and are on the same floor.
- Device movement does not collide with other devices or walls unless the product deliberately allows overlap.
- Wall addition does not collide with devices.
- Wall geometry key is canonical and unique per floor.
- Deleting a device deletes or tombstones connected links.
- Presence selections referencing missing objects are cleaned or ignored.

Do not rely on client validation for persisted correctness.

## Optimistic UI Policy

Use the session materialized view as the primary optimistic UI mechanism.

`withOptimisticUpdate` may still be useful, but only if:

- It uses ids/op ids already included in mutation args.
- It is deterministic and replay-safe.
- It does not generate ids, timestamps, random values, or history entries.
- It mirrors the same pure `applyOperation` logic as the session projection.

If this creates duplication, prefer not using Convex optimistic updates for document state and rely on the local materialized view instead.

## Drag UX

Device dragging should be local-first.

During drag:

- Keep movement local in React Flow / session transient state at 60fps.
- Validate against the visible document locally for immediate collision feedback.
- Do not write every pointer movement to durable Convex tables.
- Publish drag preview through presence at a throttled rate, around 15-30Hz.

On drag stop:

- Create one `device.patch` operation with the final position.
- Add it to pending operations immediately.
- Send it through the outbox.

Remote clients:

- See the durable position from the document snapshot.
- May see a translucent drag preview from presence while another user is dragging.
- Should not have durable state spammed by every frame.

Conflict policy for same-object drag:

- Use a soft lock/lease in presence: `editing.deviceId`, `editing.startedAt`, `editing.expiresAt`.
- If another user is dragging a device, show a visual indication and prevent accidental edits.
- Allow explicit override only if product wants it later.
- Lease must expire automatically so stale sessions never block editing.

## Undo And Redo

Undo/redo is local per session, operation-based, and creates new operations.

Undo stack item:

```ts
interface HistoryEntry {
  label: string;
  undoOperation: MapOperation;
  redoOperation?: MapOperation;
  affectedObjectIds: Array<string>;
}
```

On normal dispatch:

- Capture `snapshotBefore` from the visible document before applying the op.
- Build inverse via `buildInverseOperation(snapshotBefore, op)`.
- Push inverse to undo stack immediately if not null.
- Clear redo stack.

On undo:

- Pop an undo entry.
- Dispatch its `undoOperation` as a normal new operation with a fresh `opId`.
- At undo time, compute the redo operation from the current visible snapshot and the undo operation result.
- Push redo entry.

On redo:

- Same as undo in the other direction.

If an operation is no longer applicable because another user deleted the target:

- Do not throw.
- Do not corrupt history.
- Show a polished message such as: `Impossible d'annuler: l'element a ete modifie par un autre utilisateur.`
- Consider dropping that history entry or marking it skipped.

Required undo behavior:

- Create device undo deletes that device and its links if they were created by that action.
- Delete device undo recreates device snapshot and link snapshots with the same stable object ids if those ids are free.
- Move device undo patches position only, preserving remote edits to name/metadata.
- Wall brush undo deletes created wall ids.
- Wall erase undo recreates erased wall snapshots with stable ids if available, or fresh ids if the original ids are occupied by conflicting recreated walls. Prefer preserving ids unless it violates uniqueness.
- Batch undo reverses sub-operations in reverse order.

This matches the important Figma principle: undo/redo history is updated at undo/redo time so it does not blindly overwrite unrelated remote changes.

## Presence

Presence is ephemeral and separate from document state.

Presence can include:

```ts
interface PresenceState {
  sessionId: string;
  clientId: string;
  displayName: string;
  colorHue: number;
  floorId?: FloorObjectId;
  cursor?: Position;
  selectedObjectIds?: Array<string>;
  activeTool?: DrawTool;
  editing?:
    | {
        kind: "device.drag";
        deviceId: DeviceObjectId;
        previewPosition: Position;
        expiresAt: number;
      }
    | { kind: "wall.draw"; expiresAt: number };
  updatedAt: number;
}
```

Rules:

- Presence references application object ids, never Convex `_id`.
- Missing selected ids are ignored by renderers.
- Presence update failures should not block document editing.
- Presence TTL should remain short, around 30 seconds for liveness and shorter for edit leases.
- Presence cleanup should happen on unmount, but TTL is the real safety mechanism.

## Selection And Stale References

Zustand remains appropriate for ephemeral UI state:

- current building/floor selection
- selected device/object ids
- hovered object ids
- active tool
- highlighted ids
- debug panels
- local undo/redo stacks if they belong to the `MapDocumentProvider`

But selection must be reconciled against the visible document:

- If selected object disappears, clear selection.
- If highlighted ids disappear, remove them from highlight set.
- If drawer target disappears, close drawer with a smooth state transition.
- If presence selected id disappears, do not send it again.

Implement this reconciliation inside the session/provider boundary, not scattered across components.

## Server-Side Collision Validation

Server validation should reuse pure geometry helpers where possible.

Device move/create validation:

- Check rectangle overlap with all non-deleted devices on the floor except itself.
- Check rectangle overlap with wall collision rects.

Wall add validation:

- Canonicalize wall segment into grid blocks.
- Compute `geometryKey` for each block.
- Reject or no-op duplicate geometry keys.
- Reject wall blocks colliding with existing devices.

The client should also run the same validations for immediate feedback, but client validation is not sufficient.

## Initial Seed And No Regression From Main

The migration must preserve the richness of the previous local mock dataset.

Before merge, seed data should include:

- `Bâtiment Principal`
- `RDC`
- `Étage 1`
- the previous rack/switch/pc/wall-port devices from `main:src/mock/devices.ts`
- links equivalent to the old `metadata.connectedDeviceIds`
- realistic metadata: IP, status, model, ports, last user

Do not ship an empty first-run map unless the product explicitly changes onboarding.

## Testing Strategy

Tests are part of the implementation, not a follow-up.

Pure engine tests:

- create device is idempotent with same id and same payload
- create device with same id and different payload rejects/conflicts
- create then patch before ACK works through `applyPendingOperations`
- create then delete before ACK leaves no visible device
- patch missing device is safe no-op or structured reject
- delete missing device is idempotent no-op
- device delete removes connected links
- link create requires both endpoints
- wall add deduplicates geometry keys
- wall delete is idempotent
- batch applies in order and undo reverses order
- materialized view prevents flicker when server snapshot lags pending op

Session/outbox tests:

- operations flush sequentially in dispatch order
- ACK removes only the matching pending op
- duplicate ACK does not change state twice
- network failure keeps op pending
- retry uses same `opId`
- rejection removes pending op and exposes rejected state
- switching floor clears or scopes pending operations correctly

Convex tests:

- `mapOperations.apply` is idempotent by `opId`
- repeated mutation call returns same result
- duplicate object id create with same payload is safe
- duplicate object id create with different payload rejects
- moving into another device rejects
- moving into a wall rejects
- linking across floors rejects
- deleting a device removes/tombstones links
- presence can reference only existing floor but ignores missing object selections safely or cleans them

Integration/component tests:

- selected remote-deleted device clears drawer
- toolbar add device can immediately select/drag/delete before server ACK using stable id
- undo after remote delete does not throw
- dragging does not issue many durable operations before drop
- undo/redo buttons reflect provider history, not per-component instances

Manual QA scenarios:

- Open two browsers on same floor.
- Client A creates a device and immediately drags it before ACK.
- Client B sees it after sync; no duplicate, no jump.
- Client A deletes a device selected by Client B; Client B drawer closes cleanly.
- Both clients drag different devices; both converge.
- Both clients try to drag same device; soft lock prevents accidental conflict.
- Client A goes offline, performs one or more operations, reconnects, and pending ops either apply or reject cleanly.
- Undo a device move after Client B changes its name; name must be preserved.
- Undo delete device with links; links restore if endpoints still valid.

## Implementation Boundaries

Do not do these:

- Do not use Convex `_id` as React Flow node ids or domain ids.
- Do not generate ids inside `withOptimisticUpdate`.
- Do not have multiple independent map command hooks with their own history refs.
- Do not push undo history only after network ACK.
- Do not let server validation remain weaker than client validation.
- Do not use `any` or broad `unknown` as an escape hatch in TypeScript domain code.
- Do not introduce Yjs unless a separate decision explicitly changes the architecture.
- Do not keep zundo if it is no longer used.
- Do not remove demo richness unless replacing it with an intentional onboarding experience.

Acceptable transitional steps:

- Keep existing Convex table names while adding `objectId` fields.
- Keep Zustand for UI-only state.
- Keep current presence table but migrate references to application object ids.
- Keep current geometry/wall engine if it is made independent from Convex ids.
- Keep current UI components if they consume the new provider instead of direct commands.

## Acceptance Criteria

The architecture is implemented correctly when:

- All domain objects use stable application ids in client code.
- Convex `_id` usage is isolated to Convex/repository boundary code.
- There is exactly one active document session owner for command dispatch, pending ops, and undo/redo.
- The canvas renders a materialized document, not raw query arrays.
- Creating, editing, deleting, linking, and undoing before ACK works.
- Pending local edits never flicker backward due to server snapshots.
- Server mutation application is idempotent by `opId`.
- Server validation enforces map invariants.
- Remote deletes clean local selection/highlights/presence.
- `bun run check` passes.
- Multi-client manual QA scenarios above pass.
